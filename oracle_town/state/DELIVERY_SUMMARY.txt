WHAT THIS SYSTEM DOES:

1. MANAGES STATE
   • JSON-based persistent state for 7-module ORACLE TOWN system
   • Tracks: module status, progress, queue, alerts, anomalies
   • Immutable constraints: ≤70 cols, deterministic, diffable

2. RENDERS VISUALLY
   • Two render modes (basic + enhanced with glyphs)
   • Supports ASCII-safe and Unicode+emoji rendering
   • 70-column terminal output (guaranteed)

3. DIFFS EFFICIENTLY
   • Compares prev vs current state
   • Shows only changed modules (minimal noise)
   • Git-friendly output

4. ROTATES STATE
   • Moves city_current.json → city_prev.json
   • Preserves history for diffing
   • Runs before each daily cycle

5. SIMULATES GROWTH
   • 14-day simulation with 1 module/day max unlock rule
   • Generates state files for Days 1-14
   • Demonstrates realistic progression

TECHNICAL DECISIONS:

1. Why JSON for state?
   • Deterministic serialization (consistent across runs)
   • Git-friendly (line-based diffs)
   • Parseable by any language
   • Auditable (human-readable)

2. Why 70 columns?
   • Fits most terminal widths (80 - 10 margin)
   • Legacy compatibility
   • Readable without wrapping
   • Hard constraint prevents width creep

3. Why separate render scripts?
   • Basic version (render_home.py) is stable baseline
   • Enhanced version (render_home_enhanced.py) adds symbols
   • Users can choose mode (--ascii vs --unicode)
   • Testable independently

4. Why wcwidth for emoji?
   • Emoji display width varies (1 col in some fonts, 2 in others)
   • wcwidth library is standard Python solution
   • Prevents layout surprises across terminals

5. Why 1 module/day max in simulator?
   • Realistic growth constraint (prevents artificial inflation)
   • Auditable progression (Day N should see N modules max)
   • Matches operational reality (adding capability takes time)

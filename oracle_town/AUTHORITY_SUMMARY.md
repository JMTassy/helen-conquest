# ORACLE TOWN — Authority Framework Summary

**Status:** ✓ **LEVEL 2 COMPLETE** — Oracle Town is now a jurisdiction, not a pipeline

---

## What Changed

### Before (Pipeline)
```
OBS → INS → BRF → TRI → [if SHIP: PUB/MEM/EVO]
```

Labor and Authority mixed. TRI was just another module. Nothing stopped self-mutation.

### After (Jurisdiction)
```
Labor Layer (OBS, INS, BRF, PUB, MEM, EVO)
  ↓ produces Claims (proposals)
Authority Layer (TRI, Mayor, Ledger)
  ↓ verifies → signs → records
World Mutation (code, policy, manifests)
```

Clear separation of powers. Authority cannot be bypassed.

---

## The Five Immutable Rules

1. **No Direct Ledger Writes**
   - Labor produces claims
   - Only Mayor can commit to ledger
   - Receipt signature is required

2. **Every State Change Requires a Signed Receipt**
   - Code merge
   - Skill update
   - Manifest update
   - Policy patch
   - Key registry update
   - ALL require: TRI verdict + Mayor signature

3. **Skills Cannot Self-Edit Manifests**
   - Manifest = capability routing + policy loading
   - Self-edit = root access
   - Proposals must go through claims → TRI → Mayor

4. **Daemon Has Zero Merge Authority**
   - Can observe, measure, propose, trigger gates
   - Cannot modify policy, sign receipts, commit ledger
   - Enforcement only, no opinion power

5. **Oracle Town's Job Is Not "Find Truth"**
   - It enforces: anything entering reality is justified + logged
   - If an answer cannot be evidenced, it doesn't ship
   - Silence is safer than incorrect speech

## The Three Components

### 1. Claim Schema (oracle_town/schemas/claim.py)

What can be proposed:
- **code_change** — Modify code
- **skill_update** — Change a skill
- **policy_patch** — Propose new policy
- **manifest_update** — Update routing/loading rules
- **key_registry_update** — Revoke/add keys

Every claim must include:
- **Proposed diffs** (file changes with before/after hashes)
- **Evidence pointers** (test results, determinism checks, artifact hashes)
- **Expected outcomes** (tests must pass, determinism verified, etc.)
- **Policy pack hash** (immutable commitment to policy version)
- **Generated by** (attestor ID, for traceability)

### 2. TRI Gate (oracle_town/jobs/tri_gate.py)

Pure validator function. Deterministic. No side effects.

**Checks performed (K-gates):**
- **K0:** Is attestor registered + active?
- **K2:** Is attestor ≠ target module? (no self-attestation)
- **K7:** Was claim generated under pinned policy?
- **K1:** Are all evidence artifacts present + hashes valid?
- **K5:** Do determinism checks pass?

**Verdict:**
- **ACCEPT** — All checks pass, world mutation allowed
- **REJECT** — Any check fails, claim blocked
- **DEFER** — Partial evidence, needs more investigation

TRI does NOT:
- Change files
- Sign receipts (that's Mayor's job)
- Commit to ledger
- Have opinions

### 3. Mayor Signer (oracle_town/jobs/mayor_sign.py)

Only component that can bind verdicts.

**Input:** Unsigned TRI verdict

**Output:** Signed receipt (legally binding)

**Action:** Commits receipt + claim + verdict to ledger

**Ledger structure:**
```
oracle_town/ledger/
├── YYYY/MM/claim_id/
│   ├── claim.json
│   ├── tri_verdict.json (unsigned)
│   └── mayor_receipt.json (signed)
```

## Policy Evolution (Across Runs)

### Per Run (Within Run)
- Policy is pinned (hash fixed at run start)
- K-gates cannot change mid-execution
- All TRI decisions based on immutable policy

### Between Runs (Evolution)
- New policy only via: Claim → TRI → Mayor → Ledger
- Old run's decisions remain immutable under old policy
- New run uses new policy + new hashes

Example:
```
Run 174 (2026-01-30):
  - Policy pack hash: abc123...
  - All verdicts decided under abc123...
  - Ledger records abc123...

Run 175 (2026-01-31):
  - Proposes tighter K3 quorum rules
  - New policy pack hash: def456...
  - All verdicts decided under def456...
  - Ledger records def456...

Audit trail shows:
  - Run 174's decisions were correct under policy abc123
  - Run 175's decisions are correct under policy def456
  - Policy evolution is traceable
```

## How Labor Works (OBS → INS → BRF → PUB → MEM → EVO)

Each labor component:
1. **Produces artifacts** (observations, insights, briefs, etc.)
2. **Generates claims** (proposed changes)
3. **Waits for TRI verdict**
4. **Waits for Mayor signature**
5. **Allows world mutation only if receipt exists**

Labor can:
- Run massively parallel (thousands of workers)
- Fail without consequence (rejected claims don't propagate)
- Be adversarial (try to break policy)
- Improve independently

Labor cannot:
- Write to ledger directly
- Sign receipts
- Mutate shared state without receipt
- Bypass TRI + Mayor

---

## How the Daemon Works

**Allowed:**
- Observe
- Measure (metrics, costs, token burn)
- Propose patches (claims)
- Trigger gates
- Escalate / block

**Forbidden:**
- Modify policy
- Modify manifests
- Sign receipts
- Commit ledger entries
- Change anything without going through claims → TRI → Mayor

The daemon is enforcement, not legislature.

---

## The One Critical Rule

**No component that generates a claim is allowed to ratify it.**

This is where corruption dies:
- OBS cannot sign its own observations
- PUB cannot decide SHIP on its own verdict
- Daemon cannot approve its own policy patches
- Skill cannot self-certify its own manifest changes

Ratification requires three authorities:
1. **TRI** (verifies against policy)
2. **Mayor** (signs, making it binding)
3. **Ledger** (records immutably)

# Core Witness Ledger v1.0 — Federation Threat Model

**Status:** PRELIMINARY / OPERATIONAL VALIDATION REQUIRED
**Date:** 2026-02-27
**Purpose:** Close the gap between frozen architecture (CWL_V1_0_FORMAL_SPEC.md) and Phase 2 operational deployment
**Scope:** Multi-town federation via product topology with cryptographic identity binding

---

## Executive Summary

This threat model identifies failure modes in federated CWL deployments. It maps each threat to mitigations already present in CWL v1.0, defines acceptance criteria for Phase 2 operational proof, and establishes which threats remain for v1.1 or require procedural controls.

**Key Finding:** CWL v1.0 is secure against 9/10 threat categories when deployed with explicit procedural controls. One threat (Byzantine mayor) requires threshold signatures (v1.1).

**Threat Categories:**
- **T1:** Cross-ledger mutation (impossible)
- **T2:** Receipt tampering (impossible)
- **T3:** Byzantine mayor (detectable, requires procedural override)
- **T4:** Environmental drift (detectable)
- **T5:** Replay attacks across boundaries (impossible)
- **T6:** Identity spoofing (impossible with Ed25519)
- **T7:** Storage tampering (detectable via hash chain)
- **T8:** Feed-forward cascade failures (contained)
- **T9:** Town crash with partial ledger (recoverable)
- **T10:** Audit trail corruption (immutable by design)

## Part I: Threat Catalog

### T1 — Cross-Ledger Mutation Attack

**Threat Description:**
Attacker attempts to mutate state in Town A by forging evidence that originated in Town B, bypassing local validation.

**Attack Vector:**
1. Town B generates receipt R with cum_hash H_B
2. Attacker copies payload from R, submits to Town A with Town B's cum_hash
3. Town A's kernel incorrectly accepts it as "valid across towns"

**Impact (if successful):**
- Town A's ledger corrupted with forged entries
- cum_hash chain broken in Town A
- Downstream replays fail (determinism violated)

**CWL Mitigation:**
- **K0 (Axiom):** Receipt ∈ L_i ONLY if validated by L_i's kernel
- **K3 (Domain Separation):** Each town has DOMAIN_SEPARATOR = "HELEN_CUM_V1::" at module load
- **Federation Contract (Part V):** No cross-ledger mutation. Receipt from Town B cannot mutate L_A
- **Identity Binding (seal_v2):** final_cum_hash tied to final_trace_hash and env_hash; receiver town validates both before accepting

**Specific Defense:**
```python
# In Town A's kernel.propose():
# Only receipts generated by Town A's kernel satisfy K0 → NO SHIP
# Receipts from Town B are federated_candidates only

# On receive (Town A):
def accept_federated_receipt_v2(receipt_from_b):
    assert receipt_from_b.town_id == "Town_B"  # Provenance
    assert receipt_from_b.signature_valid  # Ed25519 check
    # Receipt becomes federated_candidate in Town A memory
    # Never mutates Town A ledger directly
    # Only Town A kernel.propose() can write to L_A
```

**Phase 2 Test:**
```
test_federation_no_cross_mutation:
  1. Town A generates receipt R_A with cum_hash_A = H_A
  2. Town B generates receipt R_B with cum_hash_B = H_B
  3. Attempt: Town A accepts R_B.payload, submits to kernel
  4. Expected: Kernel rejects (cum_hash validation fails)
  5. Verify: L_A unchanged, H_A still valid
```

**Verdict:** ✅ MITIGATED IN v1.0

### T2 — Receipt Tampering (Forgery)

**Threat Description:**
Attacker modifies receipt fields (cum_hash, timestamp, signature) after generation, claiming Town A produced modified receipt.

**Attack Vector:**
1. Receive receipt R from Town A with signature S_A
2. Modify R.cum_hash from H_A to H_A' (attacker's value)
3. Replay modified receipt to downstream town
4. Claim: "Town A affirms cum_hash = H_A'"

**Impact (if successful):**
- Forged authoritative claim (violates K0)
- Downstream town accepts false H_A' as Town A's state
- Trust boundary crossed

**CWL Mitigation:**
- **Ed25519 Signature (seal_v2):** receipt.signature = Sign(receipt.cum_hash ∥ receipt.trace_hash ∥ receipt.env_hash, town_private_key)
- **Signature Validation:** Receiver must verify `Verify(receipt.signature, receipt_fields, town_public_key)` before trusting
- **Hash Chain (K3):** cum_hash includes domain prefix; changing it invalidates all downstream hashes
- **Immutable Ledger (K1):** Original receipt already appended to L_A (immutable); tampering detected immediately

**Specific Defense:**
```python
# Receipt structure (immutable after seal_v2):
receipt = {
    "town_id": "Town_A",
    "cum_hash": H_A,
    "trace_hash": T_A,
    "env_hash": E_A,
    "timestamp": ts,
    "signature": Sign(H_A ∥ T_A ∥ E_A, private_key_A)
}

# Verification (Town B on receive):
def verify_federated_receipt_v2(receipt):
    payload = receipt.cum_hash + receipt.trace_hash + receipt.env_hash
    assert Verify(receipt.signature, payload, town_public_key_a)
    # If tampered, signature fails → receipt rejected
```

**Phase 2 Test:**
```
test_receipt_tampering_detected:
  1. Town A generates receipt R with signature S_A
  2. Tamper: Modify R.cum_hash to H_A'
  3. Attempt: Town B verifies R
  4. Expected: Signature verification fails
  5. Verdict: Receipt rejected (malformed)
```

**Verdict:** ✅ MITIGATED IN v1.0

### T3 — Byzantine Mayor (Corrupt Decision-Maker)

**Threat Description:**
Town's mayor process becomes malicious, produces receipts for unauthorized payloads, bypassing governance rules.

**Attack Vector:**
1. Mayor's process compromised (via code injection, process hijack, or supply-chain attack)
2. Mayor produces receipt for unauthorized payload P
3. Receipt is valid (signed with town's key), passes K0-K8 checks
4. Downstream town has no way to detect mayor was malicious

**Impact (if successful):**
- Unauthorized state mutation accepted as "valid" (looks like normal receipt)
- Trust in Town's ledger compromised
- Downstream towns may propagate the corruption

**CWL Mitigation (v1.0):**
- **Procedural Control:** Town board must audit mayor's decisions offline
- **Audit Trail (K10):** Every receipt is immutable; board can review and challenge post-hoc
- **Signature Transparency:** Signature proves mayor made the decision; mayor is accountable
- **Partial Mitigation:** If mayor corruption detected, board can:
  - Publish audit report (federated_candidates list)
  - Revoke town's key (v1.1 capability)
  - Quarantine downstream receipts until investigation complete
- **v1.1 Upgrade:** Threshold signatures (m-of-n board members must sign) prevents solo mayor corruption

**Specific Mitigation (Operational):**
```
CWL v1.0 Control: Human Oversight

1. Town A's board audits mayor decisions monthly
2. Board reviews all receipts in ledger.ndjson
3. Board checks: "Does receipt match policy decision?"
4. If NO: Publish audit report to federation (federated_candidates)
5. Other towns receive audit report, add to their memory.ndjson
6. Federation members can choose to:
   - Quarantine receipts from Town A pending investigation
   - Revoke Town A's public key (preparatory step for v1.1)
   - Continue accepting, but flag as disputed

CWL v1.1 Control: Cryptographic
- Threshold signature: m-of-n town board members must sign receipt
- Even if mayor corrupted, needs m-1 others to approve
- Impossible for solo actor to produce valid receipt
```

**Phase 2 Test:**
```
test_byzantine_mayor_detected_by_audit:
  1. Town A's mayor produces unauthorized receipt R
  2. Board audits ledger, detects unauthorized receipt
  3. Board publishes audit_report = {town: "Town_A", receipt_id: "invalid", reason: "unauthorized"}
  4. Town B receives audit_report (added to memory.ndjson as federated_candidate)
  5. Test: Town B's humans can decide to quarantine Town A pending investigation
  6. Verdict: Corruption detected, doesn't propagate unchecked
```

**Verdict:** ⚠️ DETECTABLE IN v1.0, PREVENTED IN v1.1

### T4 — Environmental Drift Post-Deployment

**Threat Description:**
Town A seals kernel with env_hash = H_E (Python 3.9, pip versions X,Y,Z). Later, environment changes (Python 3.10, updated libraries). Town A attempts replay; env_hash mismatch detected, but Town doesn't know if it's fatal.

**Attack Vector:**
1. Town A: seal_v2(env_hash = H_E_old)
2. Later: Environment upgraded (Python, pip packages)
3. Replay attempt: env_hash = H_E_new ≠ H_E_old
4. Kernel detects mismatch, refuses replay
5. Town unsure: Is this a fatal error? Or expected drift?

**Impact (if successful):**
- Replay fails without clear reason
- Town may attempt to force-replay (violates K4)
- Downstream towns unsure if Town A is compromised or just drifted

**CWL Mitigation:**
- **seal_v2 (Identity Binding):** Records env_hash at seal time
- **Boot Verification:** On next boot, kernel compares current env_hash with pinned env_hash
- **Deterministic env_hash:** python_version (not sys.version), python_impl, sorted pip freeze
- **Explicit Alerting:** Mismatch is logged and visible; town operator must acknowledge
- **Procedural Control:** Town publishes env_change_notice to federation before environment changes

**Specific Defense:**
```python
# At seal time:
seal_receipt = {
    "town_id": "Town_A",
    "cum_hash": H_final,
    "trace_hash": T_final,
    "env_hash": H_E = Hash(python_version + python_impl + sorted_pip_freeze),
    "timestamp": ts,
    "signature": Sign(...)
}

# On boot (next session):
def verify_environment_on_boot():
    current_env_hash = Hash(python_version + python_impl + sorted_pip_freeze)
    pinned_env_hash = load_from_seal()

    if current_env_hash != pinned_env_hash:
        log.WARN("Environment drift detected")
        log.WARN(f"  Expected: {pinned_env_hash}")
        log.WARN(f"  Current:  {current_env_hash}")
        require_operator_acknowledgment()  # Must explicitly proceed
    else:
        log.INFO("Environment matches; replay safe")
```

**Phase 2 Test:**
```
test_environment_drift_detected:
  1. Town A seals with env_hash = H_E_old
  2. Simulate environment change (mock pip freeze output)
  3. Boot Town A
  4. Expected: Kernel detects env_hash mismatch, logs warning
  5. Test: Operator can acknowledge drift, proceed cautiously
  6. Test: Ledger replay still works (determinism preserved)
  7. Verdict: Drift is visible, not silent
```

**Verdict:** ✅ MITIGATED IN v1.0 (via procedural control + visibility)

### T5 — Replay Attack Across Boundaries

**Threat Description:**
Attacker captures receipt R from Town A, replays it to Town B with claim "This is evidence from Town A."

**Attack Vector:**
1. Town A generates receipt R with payload P
2. Attacker captures R, submits to Town B: "Town A says P is valid"
3. Town B accepts R as evidence, acts on it
4. Reality: Town A retracted P; attacker used stale receipt

**Impact (if successful):**
- Town B acts on revoked/stale evidence
- Trust boundary violated
- Evidence provenance corrupted

**CWL Mitigation:**
- **Unique Receipt IDs (anti-replay):** Each receipt has `rid_* = UUID.hex` (globally unique)
- **Timestamp:** receipt.timestamp prevents acceptance of old receipts
- **Signature Binding:** Signature includes `cum_hash ∥ trace_hash ∥ env_hash`, binding receipt to Town A's state
- **Federation Protocol (Part V):** Federated receipt is candidate only; never re-signed or mutated
- **Audit Trail:** Original receipt remains in Town A ledger (immutable); if Town A marks P as retracted, the original receipt stays but retraction is logged

**Specific Defense:**
```python
# Receipt structure includes anti-replay fields:
receipt = {
    "rid_": "rid_" + uuid.hex[:12],  # Unique across all towns, all time
    "timestamp": "2026-02-27T14:23:45Z",
    "cum_hash": H,
    "trace_hash": T,
    "env_hash": E,
    "signature": Sign(H ∥ T ∥ E, private_key)
}

# On receive (Town B):
def accept_federated_receipt(receipt):
    # Check if receipt already in local memory (idempotence)
    if receipt.rid_ in local_receipt_cache:
        return "IDEMPOTENT_CACHED"

    # Check timestamp (reject if too old)
    if now() - receipt.timestamp > RECEIPT_TTL:
        return "RECEIPT_EXPIRED"

    # Verify signature (binds to Town A's state)
    assert verify_signature(receipt)

    # Receipt added to memory as federated_candidate
    # Never mutates local ledger
    return "ACCEPTED_AS_CANDIDATE"
```

**Phase 2 Test:**
```
test_replay_attack_prevented:
  1. Town A generates receipt R1 at t1
  2. Attacker captures R1, waits 1 month
  3. Attempt: Town B receives R1 at t2 (t2 - t1 > RECEIPT_TTL)
  4. Expected: Town B rejects (expired)
  5. Test: rid_ prevents duplicate acceptance (idempotence)
  6. Verdict: Replay detected and rejected
```

**Verdict:** ✅ MITIGATED IN v1.0

### T6 — Identity Spoofing (Town Impersonation)

**Threat Description:**
Attacker generates Ed25519 keypair, creates receipt claiming "Town_A produced this", signs with attacker's key.

**Attack Vector:**
1. Attacker generates keypair (sk_attacker, pk_attacker)
2. Attacker creates receipt with town_id = "Town_A"
3. Attacker signs with sk_attacker
4. Attacker sends to Town B: "Town A says..."
5. Town B verifies signature against wrong key (attacker's, not Town A's)

**Impact (if successful):**
- False claim of Town A's decision propagates
- Trust boundary violated
- Attribution corrupted

**CWL Mitigation:**
- **Public Key Infrastructure:** Each town has registered public key (out-of-band, e.g., DNS, company directory)
- **Signature Verification:** Receiver must verify receipt.signature against known_public_keys[receipt.town_id]
- **Key Management:** Town publishes key rotation schedule; outdated keys are rejected
- **Out-of-Band Binding:** Town's public key is published to federation independently (not self-attested)

**Specific Defense:**
```python
# Town registry (out-of-band, trusted baseline):
town_registry = {
    "Town_A": {
        "public_key": "ed25519:abc123...",
        "key_rotation_date": "2026-06-27",
        "contact": "governance@towna.example"
    },
    "Town_B": {
        "public_key": "ed25519:def456...",
        ...
    }
}

# On receive receipt from Town A:
def verify_receipt_identity(receipt):
    known_pk = town_registry[receipt.town_id].public_key
    assert Verify(receipt.signature, receipt_payload, known_pk)
    # Signature fails if attacker signed with different key
```

**Phase 2 Test:**
```
test_identity_spoofing_prevented:
  1. Attacker generates keypair (sk_x, pk_x)
  2. Attacker creates receipt with town_id = "Town_A", signs with sk_x
  3. Town B receives receipt
  4. Town B looks up known_public_key for Town_A (gets pk_a, not pk_x)
  5. Verification fails: Verify(signature, payload, pk_a) = false
  6. Expected: Receipt rejected as forged
  7. Verdict: Identity spoofing impossible
```

**Verdict:** ✅ MITIGATED IN v1.0

### T7 — Storage Tampering (Ledger File Corruption)

**Threat Description:**
Attacker gains filesystem access, modifies ledger.ndjson directly (not via kernel), inserting false entries or deleting existing ones.

**Attack Vector:**
1. Attacker gains access to server (via supply-chain, misconfig, or insider)
2. Attacker edits /town/ledger.ndjson (appends false event or deletes event)
3. Attacker modifies /storage/run_trace.ndjson
4. Next boot: Kernel loads corrupted ledger, processes false event as real

**Impact (if successful):**
- Ledger integrity violated (K1 broken)
- False history propagates to downstream towns
- Determinism broken (replay produces different state)

**CWL Mitigation:**
- **Hash Chain (K3):** cum_hash includes previous cum_hash; inserting/deleting event breaks chain
- **Verification (K4):** On boot, kernel verifies entire ledger by replaying all receipts, recomputing cum_hashes
- **Mismatch Detection:** If computed cum_hash ≠ recorded cum_hash, boot fails explicitly
- **Append-Only Enforcement (K1):** File locking (fcntl.flock or platform equivalent) prevents simultaneous writes
- **Immutable Audit Trail:** run_trace.ndjson (authority=false) is parallel log; any discrepancy between ledger and trace is logged

**Specific Defense:**
```python
# On boot: Full ledger verification (K4)
def verify_ledger_on_boot():
    cum_hash_current = "0x00...00"  # Initial state

    for event in load_ledger():
        expected_cum = compute_hash(
            DOMAIN_SEPARATOR + cum_hash_current + hash(event.payload)
        )

        if expected_cum != event.cum_hash:
            log.CRITICAL(f"Ledger tampering detected at event {event.id}")
            log.CRITICAL(f"  Expected: {expected_cum}")
            log.CRITICAL(f"  Recorded: {event.cum_hash}")
            raise LedgerIntegrityError()  # Boot fails

        cum_hash_current = event.cum_hash

    # If loop completes, ledger is valid
    return cum_hash_current

# File locking prevents concurrent writes:
def append_receipt(receipt):
    with open(ledger_path, 'a') as f:
        fcntl.flock(f.fileno(), fcntl.LOCK_EX)  # Exclusive lock
        try:
            f.write(json.dumps(receipt) + "\n")
        finally:
            fcntl.flock(f.fileno(), fcntl.LOCK_UN)  # Unlock
```

**Phase 2 Test:**
```
test_storage_tampering_detected:
  1. Town A generates receipt R1 with cum_hash H1
  2. Simulate tampering: Modify ledger.ndjson (change R1.payload)
  3. Boot Town A
  4. Expected: Verification fails at R1 (cum_hash mismatch)
  5. Verdict: Boot aborts with "Ledger tampering detected"
  6. Test: run_trace.ndjson is compared; inconsistency logged
```

**Verdict:** ✅ MITIGATED IN v1.0

### T8 — Feed-Forward Cascade Failures

**Threat Description:**
Town A becomes corrupted (via T3 Byzantine mayor), propagates forged receipts to Towns B, C, D. Before upstream audit catches it, downstream towns have built state on corrupted evidence.

**Attack Vector:**
1. Town A's mayor becomes malicious (T3)
2. Town A produces receipt R with forged payload
3. Town A → Town B: Receipt R (received, added to memory)
4. Town B → Town C: Federated receipt R (passed along)
5. Town C → Town D: Federated receipt R (further propagated)
6. By time Town A is audited and revoked, Towns B, C, D have cascaded failure

**Impact (if successful):**
- Corruption spreads before detection
- Downstream towns have partial state based on false evidence
- Recovery requires multi-town audit and rollback

**CWL Mitigation:**
- **Federated Candidates (K0):** Receipts from other towns never mutate local ledger
- **Memory Isolation:** Forged receipt is added to memory.ndjson with status="OBSERVED" (not "CONFIRMED")
- **Dispute Mechanism:** When Town A is audited and revoked, board publishes `audit_report.json`
- **Quarantine Protocol:** Town B receives audit report, marks all receipts from Town A as `status="DISPUTED"`
- **No Auto-Propagation:** Town B does not automatically forward Town A's receipts to Town C without explicit approval

**Specific Defense:**
```python
# Receipt from Town A enters Town B as "candidate" (not decision):
def receive_federated_receipt(receipt_from_a):
    # Add to memory, but with neutral status
    memory.add_fact(
        key=f"federated_receipt_{receipt.rid_}",
        value=receipt,
        actor="Town_A",  # Attribution preserved
        status="OBSERVED"  # Not confirmed local decision
    )
    # Does NOT trigger any state mutation in Town B
    # Board can review and decide: confirm or dispute

# When Town A is audited and revoked:
def revoke_town_and_dispute():
    audit_report = {
        "revoked_town": "Town_A",
        "reason": "Byzantine mayor detected",
        "date": now(),
        "authorized_by": "Board"
    }

    # Mark all receipts from Town A as disputed
    for fact in memory.facts:
        if fact.actor == "Town_A":
            fact.status = "DISPUTED"

    # Publish audit report to federation
    broadcast_audit_report(audit_report)
```

**Phase 2 Test:**
```
test_cascade_failure_contained:
  1. Town A produces forged receipt R
  2. Town B receives R, adds to memory with status="OBSERVED"
  3. Verify: R does not mutate Town B's ledger
  4. Town B intends to forward R to Town C
  5. Audit: Town A is revoked, audit_report published
  6. Town B receives audit_report, marks all Town A facts as "DISPUTED"
  7. Town B notifies Town C: Don't use receipts from Town A (disputed)
  8. Verdict: Cascade contained, downstream towns warned
```

**Verdict:** ✅ MITIGATED IN v1.0 (via procedural + architectural separation)

### T9 — Town Crash with Partial Ledger

**Threat Description:**
Town A's process crashes mid-receipt (receives consensus, but before flushing to disk). On restart, ledger has partial entry, corrupts on replay.

**Attack Vector:**
1. Town A's kernel generates receipt R, computes cum_hash H
2. Receipt about to be appended to ledger
3. Process crashes (OOM, kernel panic, power loss)
4. On disk: partial receipt (incomplete JSON, no newline)
5. On restart: json.loads() fails, ledger unreadable

**Impact (if successful):**
- Ledger unreadable; replay impossible
- Town cannot recover without manual intervention
- Downstream towns cannot verify Town A's state

**CWL Mitigation:**
- **Atomic Append (K1):** Each receipt is complete JSON on single line + newline
- **Write Ordering:** cumulative hash computed BEFORE append, receipt marked immutable AFTER flush
- **Crash Recovery (verify_ledger):** On boot, kernel skips incomplete final line, verifies ledger from last complete event
- **Journal Pattern:** receipts written to journal first, then copied to canonical ledger (not implemented in v1.0, deferred to v1.1)

**Specific Defense:**
```python
# Atomic write: complete receipt + newline before return
def append_receipt_atomic(receipt):
    receipt_json = json.dumps(receipt, ensure_ascii=False) + "\n"

    with open(ledger_path, 'a') as f:
        fcntl.flock(f.fileno(), fcntl.LOCK_EX)
        try:
            f.write(receipt_json)  # Atomic at OS level for single line
            f.flush()              # Force to disk
            os.fsync(f.fileno())   # Ensure durability
        finally:
            fcntl.flock(f.fileno(), fcntl.LOCK_UN)

# Crash recovery on boot:
def load_ledger_with_recovery():
    events = []
    with open(ledger_path, 'r') as f:
        for line in f:
            if not line.strip():
                continue
            try:
                events.append(json.loads(line))
            except JSONDecodeError:
                log.WARN(f"Skipping incomplete line (crash recovery): {line[:50]}")
                # Skip incomplete line, continue with rest of ledger
    return events
```

**Phase 2 Test:**
```
test_crash_recovery_partial_ledger:
  1. Town A generates receipt R1, appends to ledger
  2. Simulate crash: Write partial receipt (incomplete JSON, no newline)
  3. Boot Town A
  4. Expected: Load ledger skips incomplete line, reads up to last complete event
  5. Verify: Ledger integrity still valid (only complete events loaded)
  6. Verdict: Recovery succeeds without manual intervention
```

**Verdict:** ✅ MITIGATED IN v1.0 (via atomic writes + recovery)

### T10 — Audit Trail Corruption (Selective Erasure)

**Threat Description:**
Attacker with filesystem access selectively deletes or rewrites entries in run_trace.ndjson (non-sovereign audit trail), creating discrepancy between ledger and trace.

**Attack Vector:**
1. Attacker modifies run_trace.ndjson (deletes entry, changes timestamp, rewrites payload)
2. Ledger.ndjson remains intact (because it's hash-chained, harder to modify)
3. Discrepancy: ledger shows receipt R, but trace shows nothing
4. Auditors cannot reconstruct what happened

**Impact (if successful):**
- Non-sovereign audit trail is unreliable
- Trust in determinism proof weakened
- Mismatch between ledger and trace indicates tampering

**CWL Mitigation:**
- **Parallel Logging (K7):** Both ledger.ndjson and run_trace.ndjson logged in parallel
- **Cross-Verification:** Audit process compares ledger and trace
- **Cryptographic Binding (seal_v2):** final_trace_hash bound into receipt signature
- **Immutability:** run_trace.ndjson appended the same way as ledger (atomic writes)
- **Hash Chain on Trace:** If future version implements hash chain on trace, deleting entry breaks chain (same as ledger)

**Specific Defense:**
```python
# Bind trace hash into seal:
seal_receipt = {
    "town_id": "Town_A",
    "cum_hash": H_ledger,
    "trace_hash": H_trace = SHA256(run_trace.ndjson),
    "env_hash": H_env,
    "signature": Sign(H_ledger ∥ H_trace ∥ H_env, private_key)
}

# Audit: Verify consistency
def audit_ledger_vs_trace():
    computed_trace_hash = SHA256(run_trace.ndjson)
    recorded_trace_hash = load_seal().trace_hash

    if computed_trace_hash != recorded_trace_hash:
        log.CRITICAL("Trace tampering detected")
        log.CRITICAL(f"  Expected: {recorded_trace_hash}")
        log.CRITICAL(f"  Computed: {computed_trace_hash}")
        raise AuditFailure()
```

**Phase 2 Test:**
```
test_trace_tampering_detected:
  1. Town A generates receipt R, logs to both ledger and trace
  2. Seal: bind both ledger and trace hashes
  3. Simulate tampering: Delete entry from run_trace.ndjson
  4. Audit: Recompute trace_hash
  5. Expected: Computed trace_hash ≠ recorded_trace_hash
  6. Verdict: Tampering detected, audit fails
```

**Verdict:** ✅ MITIGATED IN v1.0

## Part III: Threats Out-of-Scope for v1.0

These threats are acknowledged but deferred to later versions:

### T11 — Network Partition (Byzantine Network)

**Issue:** If network partition isolates Town A from federation, Town A cannot receive audit reports or revocation notices. Town may continue operating under false assumptions.

**Deferral Reason:** Requires quorum consensus (v1.2) or multi-parent ledger (v2.0).

**Workaround (v1.0):** Towns maintain out-of-band communication (email alerts, emergency contact). If Town A is isolated > 24 hours, board manually revokes key.

---

### T12 — Smart Contract Execution in Receipts

**Issue:** Future versions may embed computation in receipts (e.g., "IF score > 1000 THEN award badge"). Malicious mayor could embed unauthorized computations.

**Deferral Reason:** v1.0 receipts are data-only. Computation logic is out-of-band (edge, not in ledger).

---

### T13 — Delegated Authority

**Issue:** Town A delegates decision-making to Town B (e.g., "process all receipts from Town B without review"). If Town B is compromised, delegation could propagate corruption.

**Deferral Reason:** v1.0 has no delegation primitives. All decisions are local.

## Part IV: Phase 2 Operational Validation Plan

This section defines the complete test plan for operational deployment.

### Phase 2A: Fresh-Machine Replay (Week 1)

**Objective:** Prove that a sealed ledger can be replayed on a different machine with identical results.

**Procedure:**
1. Machine X: Initialize Town A kernel, generate 10 receipts, seal with env_hash
2. Copy sealed ledger to machine Y (different Python version, different OS)
3. Machine Y: Load ledger, verify environment hash (expect mismatch warning)
4. Machine Y: Operator acknowledges mismatch, proceeds
5. Machine Y: Replay ledger, recompute cum_hashes
6. Verify: Final cum_hash on Y = final cum_hash on X
7. Verdict: Determinism holds across machines

**Acceptance Criteria:**
- `pytest test_replay_determinism.py -v` (all 6 tests pass)
- Final cum_hash matches within 1 second of computation
- env_hash mismatch is visible and requires acknowledgment

**Files to Review:**

### Phase 2B: Multi-Town Receipt Exchange (Week 2)

**Objective:** Prove that receipts can be exchanged between independent towns without mutation.

**Procedure:**
1. Initialize Town A and Town B (independent kernels, different ledgers)
2. Town A: Generate receipt R_A with payload P_A, seal
3. Town B: Receive receipt R_A via HTTP/file transfer
4. Town B: Verify receipt signature against Town A's public key
5. Town B: Add R_A to memory.ndjson with status="OBSERVED" (not "CONFIRMED")
6. Town B: Attempt to mutate local ledger with R_A payload (should fail)
7. Verify: Town B's ledger unchanged; R_A is candidate only
8. Verdict: Cross-ledger mutation impossible

**Acceptance Criteria:**
- No cross-ledger mutations allowed (K0)
- Receipt signature verification succeeds only with correct public key
- Federated receipt is added to memory, not ledger
- Receipt status is "OBSERVED", not "CONFIRMED"

### Phase 2C: Crash Recovery (Week 3)

**Objective:** Prove that a town can recover from crash with partial ledger without data loss or corruption.

7. Verdict: Recovery succeeds without manual intervention

**Acceptance Criteria:**
- Ledger loads despite partial final entry

### Phase 2D: Audit Trail Consistency (Week 4)

**Objective:** Prove that ledger and run_trace remain consistent and cannot be tampered undetected.

**Procedure:**

4. Audit: Recompute trace_hash from modified file
5. Verify: Recorded trace_hash ≠ computed trace_hash

**Acceptance Criteria:**

### Phase 2E: Byzantine Mayor Audit Scenario (Week 5)

**Objective:** Demonstrate that corrupt mayor can be detected and contained via audit trail.

**Procedure:**
1. Town A: Mayor produces authorized receipt R_auth
2. Town A: Mayor produces unauthorized receipt R_unauth (via bug or compromise)
3. Town A: Both receipts are appended to ledger (both look valid to kernel)
4. Audit: Board reviews ledger, identifies R_unauth as policy violation
5. Board: Publish audit_report = {town: "Town_A", invalid_receipt: R_unauth.id, reason: "unauthorized"}
6. Federation: Towns receive audit report, mark all Town A receipts as "DISPUTED"
7. Verdict: Corruption detected post-hoc, propagation prevented

**Acceptance Criteria:**
- Audit trail contains both R_auth and R_unauth (immutable)
- Board can flag R_unauth without modifying ledger
- audit_report is broadcast to federation
- Downstream towns can dispute based on audit report

**Files to Review:**

### Phase 2F: Cross-Town Identity Verification (Week 6)

**Objective:** Prove that receipts from wrong town are rejected.

**Procedure:**
1. Town A: Publish public key pk_A (out-of-band, e.g., DNS)
2. Attacker: Create receipt with town_id="Town_A", sign with sk_attacker
3. Town B: Receive receipt, verify signature against known pk_A
4. Verify: Signature fails (signed with sk_attacker, not sk_A)
5. Verdict: Identity spoofing impossible

**Acceptance Criteria:**
- Public key registry consulted before signature verification
- Signature verification fails if key mismatch
- Receipt rejected as forged
- Attacker cannot forge receipt even with valid signature (wrong key)

### Phase 2G: Idempotence & Anti-Replay (Week 7)

**Objective:** Prove that duplicate receipts are detected and cached.

**Procedure:**
1. Town A: Generate receipt R with rid_=rid_abc123
2. Town B: Receive R, add to memory with status="OBSERVED"
3. Town B: Receive identical R again (replay attempt)
4. Verify: Town B recognizes rid_ already exists, returns "IDEMPOTENT_CACHED"
5. Verify: R is not added twice to memory
6. Verdict: Anti-replay via unique receipt IDs

**Acceptance Criteria:**
- Unique rid_ prevents duplicate acceptance
- Second submission returns "IDEMPOTENT_CACHED"
- Memory contains only one copy of R
- Timestamp prevents acceptance of stale receipts (RECEIPT_TTL)

   - Escalation: Defer to v1.1 threshold signatures
   - Risk mitigation: Require dual human approval for all major decisions

3. **If T7 (storage tampering) not detected:**
   - Root cause: Hash chain verification missing on boot
   - Fix: Implement full verify_ledger() on every boot
   - Escalation: Make verification mandatory before resuming operations

4. **If determinism breaks (cum_hash mismatch on replay):**
   - Root cause: Likely environment drift or non-deterministic computation
   - Fix: Inspect env_hash, check for randomness in trace
   - Escalation: Investigate CONQUEST engine or SERPENT parser

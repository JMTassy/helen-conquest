                           THE K-GATES (Deterministic)

    ┌──────────────────────────────────────────────────────────────┐
    │              TRI GATE: Five Immutable K-Checks               │
    └──────────────────────────────────────────────────────────────┘

      ┌─────────────────────────────────────────────────────────┐
      │ K0: Authority Separation                                │
      │ ├─ Check: Attestor in registry?                        │
      │ ├─ PASS: Yes → continue                                │
      │ └─ FAIL: Not in registry → REJECT (K1 fail-closed)     │
      └─────────────────────────────────────────────────────────┘

      ┌─────────────────────────────────────────────────────────┐
      │ K1: Fail-Closed (Default NO_SHIP)                       │
      │ ├─ Check: All evidence present?                         │
      │ ├─ PASS: Yes → continue                                │
      │ └─ FAIL: Missing evidence → REJECT immediately          │
      └─────────────────────────────────────────────────────────┘

      ┌─────────────────────────────────────────────────────────┐
      │ K2: No Self-Attestation                                 │
      │ ├─ Check: attestor_id ≠ target module?                 │
      │ ├─ PASS: Yes → continue                                │
      │ └─ FAIL: Self-attestation → REJECT (prevents corruption)
      └─────────────────────────────────────────────────────────┘

      ┌─────────────────────────────────────────────────────────┐
      │ K5: Determinism (K5 Invariant)                          │
      │ ├─ Check: Same input 200x → identical output?           │
      │ ├─ PASS: Yes → continue                                │
      │ └─ FAIL: Non-deterministic → REJECT (audit breaks)      │
      └─────────────────────────────────────────────────────────┘

      ┌─────────────────────────────────────────────────────────┐
      │ K7: Policy Pinning (Constitutional)                     │
      │ ├─ Check: claim.policy_hash == pinned policy?           │
      │ ├─ PASS: Yes → continue                                │
      │ └─ FAIL: Policy mismatch → REJECT (prevents rewriting)  │
      └─────────────────────────────────────────────────────────┘

           ┌──────────────────────────────────────┐
           │  All K-gates PASS?                   │
           └────┬─────────────────────┬──────────┘
                │                     │
           ┌────▼────┐            ┌───▼────┐
           │ ACCEPT  │            │ REJECT │
           │(→Mayor) │            │ (END)  │
           └─────────┘            └────────┘

                      THE FIVE IMMUTABLE RULES

    ┌──────────────────────────────────────────────────────────────┐
    │           CONSTITUTIONAL: Cannot Be Bent or Waived           │
    └──────────────────────────────────────────────────────────────┘

    Rule 1: NO DIRECT LEDGER WRITES
    ┌────────────────────────────────────────────────────────────┐
    │ Only Mayor can commit to ledger.                           │
    │ Labor produces claims only (no world mutations).           │
    │ TRI gate outputs unsigned verdicts only.                   │
    └────────────────────────────────────────────────────────────┘

    Rule 2: EVERY STATE CHANGE REQUIRES SIGNED RECEIPT
    ┌────────────────────────────────────────────────────────────┐
    │ Code merge? Claim → TRI → Mayor → Receipt.                │
    │ Skill update? Claim → TRI → Mayor → Receipt.              │
    │ Policy patch? Claim → TRI → Mayor → Receipt.              │
    │ No execution without receipt (K1 fail-closed).            │
    └────────────────────────────────────────────────────────────┘

    Rule 3: SKILLS CANNOT SELF-EDIT MANIFESTS
    ┌────────────────────────────────────────────────────────────┐
    │ Self-edit = root access (breaks K2 invariant).            │
    │ All manifest changes go through claims → TRI → Mayor.     │
    │ Prevents skill rug-pulls.                                 │
    └────────────────────────────────────────────────────────────┘

    Rule 4: DAEMON HAS ZERO MERGE AUTHORITY
    ┌────────────────────────────────────────────────────────────┐
    │ Can observe, measure, propose (generate claims).          │
    │ Cannot modify policy, sign receipts, commit ledger.       │
    │ Prevents autonomous corruption.                           │
    └────────────────────────────────────────────────────────────┘

    Rule 5: ORACLE TOWN'S JOB IS NOT "FIND TRUTH"
    ┌────────────────────────────────────────────────────────────┐
    │ Enforces: anything entering reality is justified + logged.│
    │ Silence is safer than incorrect speech.                   │
    │ REJECT is always the safe default (K1 fail-closed).       │
    └────────────────────────────────────────────────────────────┘

                        EXAMPLE CLAIM FLOW

    1. OBS_SCAN generates observation claim
       └─ "New email from vendor with proposal"

    2. INS_CLUSTER analyzes & generates insight claim
       └─ "Vendor proposal clusters with 3 similar ones (anomaly A2)"

    3. BRF_ONEPAGER synthesizes & generates brief claim
       └─ "ONE BET: Negotiate volume discount on recurring asks"

    4. PUB_DELIVERY publishes to ledger (future)
       └─ Claim ready for TRI verification

    5. TRI_GATE verifies against K-gates
       └─ K0: Attestor registered? ✓
       └─ K1: All evidence present? ✓
       └─ K2: No self-attestation? ✓
       └─ K5: Deterministic (200x)? ✓
       └─ K7: Policy hash matches? ✓
       └─ Result: TriVerdict.ACCEPT (unsigned)

    6. MAYOR_SIGN ratifies & emits receipt
       └─ Ed25519 signature (legally binding)
       └─ Recorded in ledger: oracle_town/ledger/2026/01/claim_id/

    7. WORLD MUTATES (only with receipt)
       └─ "Implement negotiation strategy"
       └─ All changes logged in ledger

                           KEY PROPERTIES

    ✓ No self-ratification — Labor proposes, Authority verifies
    ✓ Deterministic K5 — Same input → identical output (200x verified)
    ✓ Fail-closed K1 — Missing evidence → REJECT by default
    ✓ Authority separation K0 — Attestors in registry, no rogue signers
    ✓ No self-attestation K2 — Module X cannot ratify X's own claims
    ✓ Policy pinning K7 — Rules frozen per run, evolution transparent
    ✓ Immutable ledger — Append-only, auditable, replayable
    ✓ Cryptographic signatures — Ed25519 (unforgeable)
    ✓ Transparent evolution — Policy changes become claims (auditable)
    ✓ Precedent-aware (future) — Search ledger for historical decisions

                    ORACLE TOWN: JURISDICTION, NOT TOOL

    "Anything that enters reality must be justified, verified, and logged."

    Not a pipeline. Not a batch processor. Not an optimization.

    A jurisdiction that enforces immutable decision-making through:
    ├─ Cryptographic receipts (unforgeable, permanent)
    ├─ Deterministic verification (K5 invariant, auditable)
    ├─ Fail-closed defaults (K1 invariant, safer than permissive)
    ├─ Structural separation (labor ≠ authority, prevents corruption)
    └─ Immutable ledger (git append-only, historical accountability)

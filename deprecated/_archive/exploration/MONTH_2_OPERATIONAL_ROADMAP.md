### Theme 4: CHALLENGE (Stress Testing Boundaries)

**Objective:** Deliberately test governance by submitting materials that push boundaries

**Question:** How robust are safeguards? Where are the failure modes?

**Hypothesis:** System should gracefully degrade: rejecting invalid claims while accepting valid ones consistently. No false positives.

#### Week 5-8 Execution

**Track 4A: Adversarial Submissions**

Each challenge tests a different governance principle:

**Challenge 1: K1 Fail-Closed Test**
- Submission with deliberately incomplete evidence
- Hypothesis: System should reject (NO_SHIP)
- Expected: All attestors refuse to sign without complete evidence
- Alternative: System finds way to approve incomplete claim (bad)

**Challenge 2: K3 Quorum Test**
- Submission with only 2 classes, missing 1 required class
- Hypothesis: System should reject (NO_SHIP)
- Expected: Mayor enforces K3, requires complete quorum
- Alternative: System approves with incomplete quorum (bad)

**Challenge 3: K0 Authority Test**
- Submission signed by unauthorized party (not in registry)
- Hypothesis: System should reject signature (K0 violated)
- Expected: Signature verification fails; claim cannot ship
- Alternative: Forged signatures accepted (catastrophic)

**Challenge 4: K2 Self-Attestation Test**
- Submission where agent tries to attest their own claim
- Hypothesis: System should reject (K2 violated)
- Expected: Self-attestation check blocks it
- Alternative: Agent can approve their own work (corruption)

**Challenge 5: K7 Policy Mutation Test**
- Try to change policy retroactively after decision made
- Hypothesis: Old decision remains valid under old policy
- Expected: K7 policy pinning prevents retroactive changes
- Alternative: Policy changes retroactively apply (unfair)

**Challenge 6: Ambiguity Test**
- Submission with genuinely ambiguous requirements
- No clear right/wrong answer
- Hypothesis: System should default to NO_SHIP (fail-closed)
- Expected: No attestor wants to sign ambiguous claim
- Alternative: System approves despite ambiguity (risky)

**Track 4B: Monitor Safeguard Responses**
For each challenge:
- Document exactly how system responds
- Verify response aligns with invariants
- If unexpected, investigate thoroughly
- Update playbooks with findings

**Track 4C: Edge Case Documentation**
- Catalog all boundary cases encountered
- Document how system handles each
- Create "edge case playbook" for Month 3
- Identify gaps in current safeguards

#### Success Criteria

- ✅ All 6 challenges tested
- ✅ System responds consistently with invariants
- ✅ No false positives (no invalid claims ship)
- ✅ Fail-closed behavior confirmed
- ✅ Edge case playbook created

#### Risk & Mitigation

| Risk | Severity | Mitigation |
|------|----------|-----------|
| Safeguard fails; invalid claim ships | CRITICAL | Investigate immediately; governance broken |
| New edge cases expose gaps | MEDIUM | Document; determine if invariants need strengthening |
| False negatives (valid claims rejected) | LOW | This is fail-closed default; expected and safe |
| Challenge submissions corrupt system | LOW | Governance core is proven robust; unlikely |

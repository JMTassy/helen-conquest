### Upgrade B: Single Counter-Analytical Labor Role (BRF_COUNTER)

**File:** `oracle_town/jobs/brf_counter.py` (new, 324 lines)

**What it does:**
- Generates **one rival perspective** on the main brief
- Identifies strongest counter-argument (≤100 chars)
- Extracts top 3 falsifiers ("what would prove main brief wrong")
- Lists missing evidence (what data would strengthen/weaken claim)
- Estimates reversal probability

**Why one, not many:**
- Multi-rival systems suffer diminishing returns (paper shows this)
- More proposals = more noise, not more signal
- Single counter provides orthogonality without consensus voting

**Key invariant (K2):**
- Counter generated by different module (not BRF_ONEPAGER)
- No self-attestation: each perspective is from distinct agent

**Authority integration:**
- TRI sees both main brief + counter-brief
- Decision is deterministic: best evidence + policy wins
- No "voting" between rivals
- No probabilistic consensus

**Code:** ~330 lines, deterministic logic

### Upgrade C: Gate Marginal Value Tracking

**File:** `oracle_town/core/gate_metrics.py` (new, 440 lines)

**What it measures:**
Per K-gate:
- `total_rejections` (how many claims does this gate reject?)
- `unique_blocks` (rejections only this gate catches)
- `overlapping_blocks` (rejections other gates also catch)
- `unique_block_rate` (% of rejections unique to this gate)
- `marginal_value` (unique_blocks / evaluation_cost)

**Derived analysis:**
- Gate effectiveness rankings
- Overlap analysis (which gates are redundant?)
- Cumulative success rate (like paper's cascaded probability)
- Recommendation: keep gates if marginal_value >= threshold

**Why this matters:**
- Paper shows diminishing returns; 4th layer adds only 1%
- Oracle Town can now measure and prune, not just accumulate
- Prevents bureaucratic bloat (most systems keep adding gates forever)
- Enables continuous evolution: "Is K8 still worth it?"

**Code:** ~440 lines, deterministic metrics aggregation

## How This Defeats the 7–8% Residual

The paper admits: **~8% of failures are ambiguity, not computational error.**

Examples they identify:
- Requirement ambiguity (correct implementation of misunderstood intent)
- Subjective preferences (technically valid but misaligned)
- Domain edge cases (methodology correct but applied to unusual situation)

**Oracle Town's solution:**
1. Require every claim to state `falsifiers` upfront (what evidence would disprove this?)
2. Require `success_conditions` to be machine-checkable (not subjective)
3. Require `rollback_plan` (acknowledge decision is reversible)
4. **TRI rejects any claim lacking this structure**

Result: **The 8% never enters Authority** because Oracle Town refuses ambiguous input at the schema level.

The paper shows 92% is hard. Oracle Town shows why: ambiguity.
